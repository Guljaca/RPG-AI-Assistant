# RPG Assistant — ролевая игра с локальными LLM

Скрипт позволяет играть в ролевые игры, используя даже небольшие модели, вроде **Qwen 3.5 9B** **Qwen 3.5 35B**.
**Для кого этот проект:**
Для тех, кому нужно честное мышление от нейронки с полным обоснованием тех или иных решений.
---

## 📖 Описание возможностей

- **Многоэтапная обработка** — программа делит процесс обработки запроса на несколько этапов. На каждом этапе модели даётся одна задача и минимальный `system prompt`.
- **Гибкое управление контентом** — внутри интерфейса можно создавать игровые объекты, редактировать промты и менять их очередность загрузки.
- **Не навязывает решения** — скрипт не принимает решений за модель, он только разбивает задачу и передаёт данные между шагами.
- **Общение через Tools** — модель взаимодействует с программой через механизм Tools.
- **Элементы случайности** — добавлены случайные события, чтобы генерация была интереснее.
- **Множество NPC** — программа позволяет модели обрабатывать сразу много персонажей.
- **Непредвзятость** — реакция каждого персонажа обрабатывается отдельно: модель сначала определяет, что персонаж «планирует» сделать, затем сводит их решения на подготовленную сцену и выводит итог.
- **Полная обратная связь**:
  - Интерфейс показывает, какие промты отправляются в модель.
  - Отображается весь процесс мышления модели (включая обращения к скрипту).
  - В папке `logs` хранится до 30 логов с каждым решением модели.
  - Возможность перегенерации последнего сообщения.
- **Долговременная ассоциативная память.** Модель сохраняет информацию о каждом объекте, и получает ее только если объект есть на сцене.
- ** Полный контроль над всем процессом принятия решений**: каждый шаг размышлений можно отключить и отредактировать

## ⚙️ Установка и использование

1. **Установите Python** и пропишите его в `PATH`.
2. **Установите LM Studio**.
3. **Скачайте подходящую модель** (см. требования ниже).
4. **Запустите `install_deps.bat`** — установятся зависимости Python.
5. **Запустите `run_rpg_assistant.bat`**.
6. В настройках выберите модель из списка.
7. Нажмите **«Начать игру»**.

---

## 🧠 Требования к модели

- Поддержка **Tools**
- Поддержка **Thinking**
- Желательно **Instruct**
- Желательно **Uncensored**
- Подойдёт квантизация `Q4_K_M`, `Q6` позволит чуть больше контекста.

### Рекомендуемые модели
- Настоятельно рекомендую модель: **qwen3.5-9b-claude-4.6-os-auto-variable-heretic-uncensored-thinking-max-neocode-imatrix@q6_k** Причина: все промты оптимизированы под нее.
- **Qwen 3.5 9B** — даёт хороший результат.
- Личный пример автора:  
  `qwen3.5-9b-claude-4.6-os-auto-variable-heretic-uncensored-thinking-max-neocode-imatrix@q6_k`

## ⚠️ Важное замечание по проблемам

- Если в `Thinking` стало появляться **мало деталей** — вы перегрузили модель информацией.
- Если модель не может несколько раз вызывать функции — вы перегрузили модель информацией.
- Первым делом отключите короткую память
- Нужно уменьшить длину промтов.
- Попробуйте перегенерировать сообщение
- Уменьшите количество сообщений в истории до 2-х и отключмие краткую память
- Попробуйте временно отключить историю и другие не системные промты
- Уменьшите температуру до 0.3-0.5

## СТАДИЯ 0
- Игрок отправляет сообщение

## СТАДИЯ 1 `_stage1_request_descriptions`

### Шаг 1.1 «Запрос описаний объектов»

**Модель получает:**
- Системный промт
- Назначенные промты с правой панели в соответствующем порядке.
- Сообщение от игрока: она должна воспринимать его как запланированное игроком действие, которое он попытается совершить.
- Скрипт присылает список всех доступных объектов (предметы, персонажи, локации) — id и имена.
- Скрипт присылает информацию из ассоциативной памяти об этих объектах.

**Обработка информации:**
- Модель объединяет эту информацию в обновлённый список объектов.
- Модель должна решить на основании имеющейся информации, какую сцену разыграть.
- Поняв, какая будет сцена, модель должна решить, какие промты объектов запросить у скрипта.
- Запрашивает промты объектов у скрипта специальной командой.
- Получив всю информацию, начинает обдумывать сцену, прикидывать общие очертания.
- Приняв решение, она отправляет «Список объектов с описаниями» скрипту специальной командой.

#### Упрощённая схема логики (Шаг 1.1)
1. Получение объектов
2. Запрос информации
3. Чтение информации и определение, подходят ли объекты сцене по имеющимся вводным данным
4. Вывод проверенных объектов

**Вывод:**
- `-> «Список объектов с описаниями»`

---

### Шаг 1.2 «Создание сцены»

**Модель получает:**
- Системный промт
- Назначенные промты с правой панели в соответствующем порядке.
- Сообщение от игрока: она должна воспринимать его как запланированное игроком действие, которое он попытается совершить.
- Модель получает «Список объектов с описаниями». Читает, отсеивает негодные.
- Модель создаёт сцену на основе полученной информации.
- Модель отправляет сцену скрипту специальной командой.

#### Упрощённая схема логики (Шаг 1.2)
- Ввод проверенных объектов → Создание сцены → Вывод сцены

**Вывод:**
- `-> «Описание сцены»`

---

- «Список объектов с описаниями»
- «Описание сцены»

### Обработка ошибок (Стадия 1)
- Если произошла ошибка — попытаться определить результат парсингом сообщения.
- Если за отведённые попытки решение не будет принято и скрипт возвращался со стадии 2 два раза — принять последний «Список объектов».

- Если правила бросков кубиков не предоставлены — симулирует бросок и определяет успешность исхода.
- Описывает, что происходит с игроком на основании полученных данных.
- Описание отправляет скрипту специальной командой.

**Вывод:**
- `-> «Результат действий игрока»`

## СТАДИЯ 8.1 — Проверка истории (цикличная)

На этой стадии происходит циклическая проверка соответствия ответа записям в истории.

**Входные данные:**
- Краткая память
- Ассоциативная память
- Если «Включить группировку историй: Нет» — приходит список записей истории.
- Если «Включить группировку историй: Да» — приходит список сгруппированных историй.
- Приходит «Итоговая история»
- Настройка: «Количество новых историй, с которыми будут сравниваться старые истории: 2»

**Примечания:**
- В каждый последующий цикл должно приходить изменённое на предыдущем этапе сообщение и оригинал для контроля искажений.
- История должна приходить по 2 записи на каждый этап: ответ пользователя, ответ модели.
- Нужно сообщать модели, сколько сообщений прошло между историей и текущим сообщением, чтобы она могла определить, насколько сильно информация может отличаться при внесении изменений.

### Цикл проверок

#### Цикл 1 (отдельный запрос)
1. На входе: «ассоциативная память», самая новая история или группа историй (в количестве, указанном в настройках). Указывается высокий приоритет новой истории или группы.
2. Проверяется соответствие информации сообщению (пример несоответствия: в ассоциативной памяти дверь закрыта, а в ответе она открыта).
3. Вносятся минимальные правки в финальный ответ.
4. Ответ выводится.

#### Цикл 2 (отдельный запрос)
1. На входе: «краткая память», самая новая история или группа историй. Указывается высокий приоритет новой истории или группы (в количестве, указанном в настройках).
2. Проверяется соответствие информации сообщению. Так как это самая далёкая память, ищутся только очень грубые ошибки (пример: игрок победил тролля в кулачном бою, а в сообщении его называют новичком в драках).
3. Вносятся минимальные правки в финальный ответ.
4. Ответ выводится.

#### Цикл 3 (отдельный запрос)
Модель получает пары памяти: **ответ пользователя + ответ модели** на этот запрос, а не по отдельности!

1. На входе: последняя (самая далёкая по времени), ещё не проверенная история или группа историй **И** самая новая история или группа историй (в количестве, указанном в настройках). Указывается высокий приоритет новой истории или группы.
2. Проверяется соответствие информации сообщению.
3. Вносятся минимальные правки в финальный ответ.
4. Ответ выводится.

**Цикл 3 повторяется**, пока все истории или группы истории из списка не будут проверены.  
Каждый раз модель получает новый запрос.  
Эти запросы должны логироваться и отображаться в центральном окне. Мысли — в панели Thinking, прогресс — в панели вывода сообщения.

## СТАДИЯ 8.2 — Validation (проверка качества)

**Алгоритм проверки:**
1. В сцене участвуют персонажи, которые участвовали до этого в истории, или не участвуют, но это логически обосновано.
2. Планы персонажей должны соответствовать итоговому выводу.
3. Действие игрока должно соответствовать его обработке и выводу.
4. Действия должны соответствовать результатам бросков кубиков.
5. Модель не должна подменять кубики своими значениями.
6. Проверка соответствия сеттингу из описания и истории.
7. Проверка отсутствия логических несостыковок.
8. В ответе должна быть конкретика вместо обобщений.

Незначительные недостатки пропускать можно.  
Этот шаг должен получать всю необходимую информацию для проверки.  
Если шаг одобряет историю — он её пропускает дальше.

# Логика: Сжатие памяти

## Пример настроек
- Максимальное количество отправляемой на этап памяти: **12**
- Включить группировку историй: **да**
- Количество памяти в группах истории: **5**
- Количество новых историй, с которыми будут сравниваться старые истории: **2**

В данном примере количество групп, передаваемых модели:  
`12 / 5` с округлением вверх = **3 группы**:
- «Группа историй 1» — до 5 объединённых историй
- «Группа историй 2» — до 5 объединённых историй
- «Группа историй 3» — до 2 объединённых историй (остаток)

### Процесс (по шагам)

**0. Начало игры**

**ОТВЕТ 1**
1. Игрок пишет сообщение.
2. Модель отвечает.
3. Ответ игрока и ответ модели заносятся в историю в `sessions` как «История 1».
4. Модель дублирует «Историю 1» в `session` «Группа историй 1».

**ОТВЕТ 2**
1. Игрок пишет сообщение. Модель получает на вход «Группа историй 1».
2. Модель отвечает.
3. Ответы заносятся в историю как «История 2».
4. Модель объединяет «Группа историй 1» и «История 2» в одну «Группа историй 1», сжимая информацию до фактов, например:
   - Действие игрока. Результат действий. Реакция персонажа 1 на действие. Реакция персонажа 2 на действие… Изменение объектов (локация, предмет, персонаж), вызванные этим действием.
   - Действие персонажа. Результат действий. Реакция игрока на действие. Реакция персонажа 1 на действие… Изменение объектов, вызванные этим действием.

**ОТВЕТ 3**
1. Игрок пишет сообщение.
2. Модель отвечает.
3. Запись как «История 3».
4. Модель объединяет «Группа историй 1» и «История 3» в одну «Группа историй 1» (сжатие по тому же принципу).

**ОТВЕТ 4** — аналогично  
**ОТВЕТ 5** — аналогично  

**ОТВЕТ 6**  
Создаётся новый блок памяти **«Группа историй 2»** (потому что в настройках «Количество памяти в блоках памяти: 5»).


## Общие замечания

- **Инструменты (Tools):** используются для передачи данных между моделью и скриптом на всех стадиях.
- **Повторные попытки:** при ошибках модель пытается распарсить сообщение; если после заданного числа попыток решение не принято — используется последнее допустимое значение или выполняется переход по умолчанию.
- **Системные промты:** задаются отдельно для каждой стадии и всегда добавляются последними в список промтов.
